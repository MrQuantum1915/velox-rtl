% ─── Abstract ────────────────────────────────────────────────────────────────
\begin{abstract}
This report provides a design and implementation of a real-time speed measuring system implemented in Verilog HDL and designed to run on the Digilent Nexys A7 FPGA board. The installed design has two infrared sensors as digital inputs, filters sensor noise, counts valid quadrature transitions within a fixed 0.5s observation window, and converts the result to speed in cm/s with a design-time calibration factor, and displays the result on the onboard seven-segment display. An overspeed indicator is a special red LED. The report is based purposefully on the checked-in RTL and constraints as opposed to a conceptual description alone. That is why it records the real module hierarchy, measurement scheme, display persistence scheme, pin mapping, and current verification status in the repository. Placeholders and blank spaces are provided in any place where the project team will need to insert a hardware-specific value or Vivado-generated screenshots in the future.
\end{abstract}

% ─── Section 1 ───────────────────────────────────────────────────────────────
\section{Introduction}
Motion measurement in digital hardware is an interesting project since it bridges sensor interfacing, synchronous logic, timing, data representation and formatting output in a single system. In this project, two IR sensors produce digital signals, which indicate the movement of an object. These are linked to the FPGA by the PMOD interface of the Nexys A7 board. When the signal activity is captured, it is processed completely in hardware without software.

The RTL deployed in the repository makes the project a fixed-window pulse-counting speedometer. This is a significant difference. A wide category of speed measurement systems either counts pulses on a wheel or encoder, or measures time of flight between two spatially separated sensors. The checked-in Verilog is of the latter type. It measures forward transitions of quadrature states, over a half-second time interval, and then uses a fixed scale factor to approximate speed. The report thus concentrates on the design which can be synthesized in the existing repository.

The project is also relevant in the Computer Organization and Architecture perspective since it is based on explicit modular decomposition, synchronous state observation, arithmetic scaling, display multiplexing, and timing-driven implementation on programmable logic. Practical engineering problems revealed in the work include width overflow, reset polarity, asynchronous signal sampling, and simulation and implementation artifact consistency.

\section{Project Statement and Objectives}

\subsection{Problem Statement}
The project goal is to create an FPGA-based system, which takes the input of two IR sensors, calculates the speed of a moving object based on sensor transitions, and displays the measured value in real time on the Nexys A7 board. It must be able to run predictably at the onboard 100 MHz clock, ignore small sensor glitches, and give a visual cue when the measured speed is above a specified limit.

\subsection{Project Objectives}
The specific objectives of the implemented design are listed below.
\begin{enumerate}
\item Interface two IR sensor signals to an FPGA using proper logic-level constraints.
\item Clean the raw sensor inputs before using them in measurement logic.
\item Count valid forward quadrature transitions in a fixed measurement interval.
\item Convert the measured count into speed using a calibration constant embedded in the RTL.
\item Display the result on the onboard seven-segment display.
\item Retain the last non-zero reading briefly so the user can observe the output.
\item Generate a red LED alert for overspeed conditions.
\item Provide a repository structure that can be simulated and synthesized in Vivado.
\end{enumerate}

\subsection{Report Scope}
The hardware platform, code architecture, design equations, interface mapping, planned verification flow and implementation placeholders are discussed in this report. It also logs the engineering constraints that can be seen in the repository such that the final report submitted is technically consistent with the current source code.

\section{Implemented System Summary}

\subsection{What the Repository Actually Implements}
The source file \texttt{src/speedometer\_top\_nexys.v} contains the entire synthesizable design. The top-level module instantiates six helper blocks: an IR input filter, a clock divider, a pulse counter, a speed calculator, a BCD converter, a seven-segment multiplexer, and a red LED controller. A small display-hold register block is also coded directly in the top-level module.

The measurement principle implemented in the design is summarized as follows.
\begin{enumerate}
\item The two IR channels are sampled and filtered.
\item A 26-bit counter defines a fixed observation window of 0.5\,s.
\item During the active window, a pulse counter increments for valid forward quadrature transitions in the sequence \texttt{00}$\rightarrow$\texttt{01}$\rightarrow$\texttt{11}$\rightarrow$\texttt{10}$\rightarrow$\texttt{00}.
\item At the end of the window, the pulse count is multiplied by 16 to form the speed output in cm/s.
\item The measured value is converted to decimal digits and shown on the seven-segment display.
\item A red LED is asserted when the measured speed crosses the overspeed threshold.
\end{enumerate}

\subsection{Why the Distinction Matters}
The README file contains a higher-level architectural narrative that mentions a four-stage pipeline and time-interval measurement between sensors. However, the synthesizable Verilog checked into the repository uses a pulse-counting architecture instead. For an academic report, describing a design that is not implemented would weaken the submission. This document therefore presents the pulse-counting speedometer as the principal design and treats any broader conceptual description only as background motivation.

\subsection{Functional Output of the Current Design}
The current design produces three directly visible outputs on the board.
\begin{enumerate}
\item A three-digit numerical speed readout using the onboard eight-digit seven-segment display.
\item A red LED overspeed indicator.
\item Persistent display retention of the last non-zero speed value for approximately five seconds.
\end{enumerate}

\section{Design Methodology}

\subsection{Measurement Principle}
Let $N_p$ denote the number of valid counted transitions during one measurement window, and let $T_w$ denote the measurement window. From the code,
\begin{equation}
T_w = 0.5\ \text{s}
\end{equation}
and the speed computation implemented in hardware is
\begin{equation}
v_{\text{cm/s}} = 16 \times N_p.
\end{equation}

This means the present design already assumes a fixed calibration constant. If the measured transitions correspond to an object displacement $d_p$ per counted pulse, then the generic relation is
\begin{equation}
v_{\text{cm/s}} = \frac{N_p \cdot d_p}{T_w}.
\end{equation}
By comparing with the implemented form, the current constant is equivalent to
\begin{equation}
\frac{d_p}{T_w} = 16 \quad \Rightarrow \quad d_p = 8\ \text{cm}
\end{equation}
for $T_w=0.5$\,s. In other words, the source code behaves as if one valid counted event corresponds to 8\,cm of travel. This report leaves a calibration entry blank in later sections so that the team can confirm whether that assumption matches the physical setup.

\subsection{Reset Strategy}
The reset button on the board is active high, while most internal modules use active-low reset. The highest level module thus flips the reset button to form \texttt{reset\_n}. The method is easy and suitable to a small design, but a synchronized release of the reset can be implemented in later revisions where necessary.

\subsection{Input Cleaning Strategy}
The IR channels are asynchronous with respect to the FPGA clock. The present design is not a complete two-flip-flop synchronizer, but a three-sample filter, which only changes the output when three successive identical samples are seen. This technique is a glitch filter, and offers a certain amount of practical stabilization, though a more robust final version would still be a good idea with a strict metastability-hardened synchronizer stage.

\subsection{Display Strategy}
The result of the measurement is not plotted to the display on a cycle-to-cycle basis. Rather, the highest level logic holds the latest non-zero value and retains it on-screen during five seconds. This can be applied in a physical experiment since the object can be moving fast and the user still has time to read the number displayed on the screen.

\section{RTL Architecture}

\subsection{Top-Level Integration}
The top-level module integrates all functional blocks and implements the display hold feature. It has limited inputs and outputs and is directly connected to board-level interfaces.

\placeholderfigure{1.8in}{Insert Vivado RTL schematic of the top-level design showing all instantiated blocks and signal connectivity.}

\subsection{IR Filter Module}
The \texttt{ir\_filter} block uses a 3-bit shift register and updates the filtered signal only when the register is either \texttt{111} or \texttt{000}. This effectively rejects short transients and removes bouncing disturbances on the sensor lines. It is a compact and resource-efficient hardware solution.

\subsection{Clock Divider Module}
A 0.5\,s timing reference is generated by the clock divider block. A 26-bit counter counts system clock cycles from 0 to 49{,}999{,}999. Upon reaching the terminal count, the module asserts \texttt{timer\_done} and resets the counter. This provides a stable sampling interval from the 100\,MHz master clock without introducing a new clock domain.

\subsection{Pulse Counter Module}
The pulse counter compares previous and current values of the filtered IR signals and increments the count only for valid forward quadrature transitions. Invalid transitions, repeated states, or noise are ignored. The current implementation supports only one direction, which is sufficient for unidirectional measurement.

\subsection{Speed Calculator Module}
The speed calculator performs a simple multiplication by 16. This minimizes hardware complexity by avoiding division operations. The implementation reflects a calibrated scaling factor rather than a general physical formula.

\subsection{BCD Converter}
The BCD converter extracts the hundreds, tens, and ones digits of the computed speed. Since only three digits are displayed, values greater than 999\,cm/s wrap around modulo 1000 at the display stage. Internally, speeds up to 8176\,cm/s can be stored.

\subsection{Seven-Segment Multiplexer}
The display uses time multiplexing across three digits, driven by a 17-bit divider to ensure a stable refresh rate. Both segments and anodes are active-low, consistent with Nexys A7 hardware. Note that discrepancies may exist between XDC comments and source comments; the actual RTL signal behavior should be considered authoritative.

\subsection{Red LED Controller}
The \texttt{red\_led\_controller} activates an LED when the measured speed exceeds 200\,cm/s. This threshold can be adjusted via an RTL parameter. It provides immediate visual feedback, especially when the display changes rapidly.

\subsection{Display Hold Logic}
The display hold logic is implemented in the top-level sequential block. After a valid measurement window, the computed speed is stored and held for 5\,seconds using a 30-bit counter. This improves readability without significantly increasing design complexity.

\subsection{Dataflow Summary}
The complete dataflow of the implemented system can be summarized in a compact hardware sequence:
\begin{equation}
\begin{split}
\text{IR inputs} \rightarrow \text{filtered states} \rightarrow \text{quad. count} \\
\rightarrow \text{scaled speed} \rightarrow \text{digits} \rightarrow \text{display/LED}
\end{split}
\end{equation}
This clean flow is suitable for explanation in a COA report because it shows how a physical input stream is transformed into a human-readable output through distinct digital stages.

\subsection{Timing Considerations}
The design uses only one clock domain. This simplifies timing closure and avoids inter-clock synchronization problems. The most timing-sensitive logic is modest in complexity: counters, comparators, combinational selection, and a small amount of arithmetic. Therefore, the design is expected to meet timing comfortably on a 100\,MHz Artix-7 device, although the final timing summary should still be inserted from Vivado implementation reports.

\section{Discussion}

The current design has strengths that are as follows.
The project has a number of strengths that make it suitable for a COA/FPGA demonstration. It can be grasped in a single sitting, but it has enough to demonstrate actual tradeoffs in digital design. It employs synchronous counting, arithmetic conversion, active output driving and human-visible indication. The code is also realistic: it already documents fixes concerning width and refresh rate, which implies active debugging as opposed to shallow assembly.

\subsection{Design Limitations}
The current implementation is not without its limitations and these need to be honestly mentioned in the report.
\begin{enumerate}
\item The speed equation is coded in hard by a constant multiplier and thus relies on external calibration assumptions.
\item The speed shown is restricted to three decimal places although the internal value may be more than 999.
\item The existing input cleaning technique is handy but not a formal synchronizer chain.
\item Counts only forward quadrature transitions.
\item The testbench being checked in needs to be corrected in terms of naming before its results can be reproduced.
\item Repository comments have some inconsistencies with the synthesizable behavior and cannot be mindlessly copied into the final narrative.
\end{enumerate}

\subsection{Suggested Future Scope}
In case the project is continued in a later semester or demonstration, the following would be technically significant.
\begin{enumerate}
\item Add an appropriate synchronizer before the filter to each asynchronous IR input.
\item Change the constant multiplier to a configurable calibration register or switch input.
\item Use explicit units of display like cm/s, m/s or km/h.
\item Increase dynamic range by displaying more than four digits.
\item Add full quadrature interpretation bidirectional motion detection.
\item Measure values to UART, VGA, or external memory and analyze.
\item Include a finite state machine to provide better control and diagnostics.
\end{enumerate}

\section{Conclusion}
In this report, a dual IR sensor based speed measurement system was reported in Verilog HDL on the Nexys A7 FPGA board. The architecture in the repository is a calibrated pulse-counting speedometer, rather than a generic time-of-flight system, and the report was deliberately authored to fit that implemented architecture. The system averages the two sensor inputs, counts the valid quadrature transitions within a 0.5\,s window, calculates the speed based on a fixed factor, shows the number on the seven segment display, and indicates overspeed via a special LED.

Academically, the project is a good introductory FPGA implementation due to its practical hardware design with visible outputs and distinct module boundaries. After filling in the remaining blanks with team information, the values of the exact hardware setup, and Vivado-generated screenshots, the report is to be a fully submission-ready document in the COA project.

\subsection*{A.2\ \ IR Filter, Clock Divider, and Pulse Counter}
\begin{lstlisting}
module ir_filter (
    input wire clk,
    input wire signal_in,
    output reg signal_out
);
    reg [2:0] shift;
    always @(posedge clk) begin
        shift <= {shift[1:0], signal_in};
        if (shift == 3'b111)
            signal_out <= 1;
        else if (shift == 3'b000)
            signal_out <= 0;
    end
endmodule

module pulse_counter (
    input  wire       clk,
    input  wire       reset_n,
    input  wire       ir1,
    input  wire       ir2,
    input  wire       timer_done,
    input  wire       timer_reset,
    output reg  [8:0] pulse_count_out
);
    wire [1:0] curr_state = {ir1, ir2};
    reg  [1:0] prev_state;

    always @(posedge clk or negedge reset_n) begin
        if (!reset_n) begin
            pulse_count_out <= 0;
            prev_state      <= 2'b00;
        end else begin
            if (timer_reset) begin
                pulse_count_out <= 0;
            end else if (!timer_done) begin
                case ({prev_state, curr_state})
                    4'b00_01, 4'b01_11, 4'b11_10, 4'b10_00:
                        pulse_count_out <= pulse_count_out + 1;
                    default: ;
                endcase
            end
            prev_state <= curr_state;
        end
    end
endmodule
\end{lstlisting}

module speed_calculator (
    input  wire [8:0]  pulse_count_in,
    output reg  [12:0] speed_cm_s_out
);
    always @(*) begin
        speed_cm_s_out = pulse_count_in * 13'd16;
    end
endmodule

module bcd_to_digits (
    input  wire [12:0] speed_in,
    output reg  [3:0]  hundreds_out,
    output reg  [3:0]  tens_out,
    output reg  [3:0]  ones_out
);
    always @(*) begin
        hundreds_out = (speed_in % 1000) / 100;
        tens_out     = (speed_in % 100)  / 10;
        ones_out     =  speed_in % 10;
    end
endmodule

module red_led_controller (
    input  wire [12:0] speed_in,
    output reg         red_led_out
);
    parameter SPEED_LIMIT = 13'd200;
    always @(*) begin
        red_led_out = (speed_in > SPEED_LIMIT) ? 1'b1 : 1'b0;
    end
endmodule
\end{lstlisting}
